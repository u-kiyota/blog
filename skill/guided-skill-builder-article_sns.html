CLAUDE SKILL 紹介

スキルは作れる。でも「二度目に動かしたら壊れた」問題、ありませんか？

Claude の skill-creator を使えば、業務を自動化するスキルはサッと作れます。けれど数週間後にもう一度動かすと、なぜか二重登録される。同僚に渡すと環境が違って動かない。履歴が膨らんで遅くなる——。guided-skill-builder は、その「作って終わり」を卒業させるためのラッパースキルです。

ひとことで言うと——「何をするか」は標準の skill-creator に任せ、「どう設計するか」を運用目線の設計原則で縛る。 だから、再実行しても壊れず・人に引き継げて・使いながら育つスキルが手に入ります。

01ラッパーって、どういうこと？

skill-creator の「ドラフト → テスト → 評価 → 改善 → パッケージ」という王道プロセスは、そのまま使います。guided-skill-builder はその各ステップに設計ガイドラインを差し込むだけ。車輪の再発明はしません。役割分担はこうです。

skill-creator標準の作成プロセス
＋
設計原則を注入運用に耐える設計へ
→
育つスキル壊れない・引き継げる

02注入される「設計原則」がすごい

中身は A〜F の6グループにわたる原則集で、それぞれに「なぜそうするのか」が添えられています。丸暗記のルールではなく、目の前のスキルに合わせて賢く適用するための考え方です。抜粋するとこんな感じ。

A ／ コア設計

毎回“考え直さない”仕組み

判断は対応表として外部化、反復処理は scripts/ にコード化、読むファイルは分割（progressive disclosure）。安定・高速・低コストに。

A ／ 引き継ぎ・分割

途中から再開できる

状態を引継ぎファイルに書き出し、業務は段に分けて中間ファイルでバトンタッチ。重い工程はサブエージェントに委譲してコンテキストを軽く保つ。

B ／ 安全・可視化

壊す前に必ず止まる

登録・更新・削除は「内容を提示 → 承認 → 実行」。基本は下書き止まり。結果は見やすい HTML で可視化するのが推奨。

C ／ 公式コア原則

どの環境でも動く

他スキルと衝突しない composability、環境差を吸収する portability、そして起動精度を決める description の質。Anthropic 公式の勘所を反映。

D ／ 追加原則

再実行に強い

同じ入力で二重登録しない冪等性、着手前の前提チェックで fail-early、変更履歴、そして共有時の固有名詞の匿名化まで。

F ／ エージェント運用

記憶し、先回りし、育つ

状態を跨ぐ記憶設計、長時間タスクの文脈管理、過去ログを再整理して先回りする仕組み、そして“感想”でなく計測で回す自己改善ループ。

032枚の「ゲート」で品質を守る

guided-skill-builder の核心は、工程の節目に置く2種類のゲートです。ビルダー自身の作成プロセスにも、生成されるスキルの手順にも、両方に組み込まれます。

  • 品質ゲート：ドラフト後・梱包前に別エージェントが独立レビュー。作る人と検査する人を分けるから見落としが減る。機械判定できる部分は検証コードで合否を出す。
  • 承認ゲート：テスト確定前・外部システムへ書き込む前・最終パッケージ前に、人が内容を見て承認。見た目の判断が要る成果物は目視確認を挟む。

04使い方は、話しかけるだけ

難しい呼び出しは不要です。スキルを作りたい／改善したい意図が伝われば起動します。

あなた ▸ この請求書チェックの作業をスキルにして
あなた ▸ あのスキルの精度を上げたい
あなた ▸ ○○を自動化するスキルがほしい

あとは前提（環境・使う MCP・定期実行の要否）を確認しながら、設計原則を効かせて作成 → ゲートを通して → .skill として提示、まで伴走します。

05Before / After

▼ ふつうに作ったスキル

  • 二度目の実行で二重登録
  • 環境が変わると動かない
  • 履歴が膨らんで遅い・不安定
  • いきなり本番データを更新して事故
  • 作った本人しか直せない

▲ guided-skill-builder で作ったスキル

  • 冪等 — 何度流しても安全
  • 環境差を吸収して動く
  • 記憶を重み付けして軽快なまま
  • 更新前に提示 → 承認で止まる
  • 引継ぎ・変更履歴つきでチームで育つ

「一度作って終わり」から、「使いながら育てる」へ。
